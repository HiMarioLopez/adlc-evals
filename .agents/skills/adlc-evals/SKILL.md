# ADLC Evals — Project Skill

AI-driven technical comparison reports with human validation. This skill is the single entry point for any agent or contributor working inside this repo.

## First Rule: Source of Truth

The **site's TypeScript data** (`site/data/reports/{vendor}/*.ts`) is the source of truth. The markdown reports in `generated-reports/` are downstream artifacts. Every sharded report includes this note in its footer:

> This markdown mirrors the curated, human-validated content published at the live site above. Where the two drift, **the site wins** — this file is regenerated from site data during each refresh.

Act accordingly. Never edit markdown to override site data. When the site changes, the markdown must be updated too, not the other way around.

## Second Rule: Reports and Prompts Are Directories

The large reports and meta-prompts are **sharded** into directories (one `.md` file per `##` section) because edit tooling chokes on 1,000+ line files. A shard dir contains:

- `00-header.md` — H1 + preamble
- `NN-<slug>.md` — one file per top-level `##` section, zero-padded, sorted in reading order
- `index.md` — TOC auto-generated by the sharder

**Never edit a shard without knowing which one holds the section you want.** Always `ls` the directory or read its `index.md` first.

**Byte-identical reconstruction** is a hard guarantee. `cat NN-*.md` rebuilds the original flat file exactly.

## The Pipeline (7 Steps)

```
A. Create/Update Meta-Prompt  →  B. Generate Report  →  C. Refine
→  D. Human Review  →  E. Update Site  →  F. Deploy  →  G. Iterate
```

Full step-by-step playbooks: see `references/pipeline-playbook.md`.

The critical agent-executable moments:

| Step | Artifact | Agent action |
|------|----------|--------------|
| A | `meta-research-prompts/{Vendor}-Base-Research-Prompt/` | Edit shards directly; use Appendix B (shard `15-…` / `16-…`) for self-refresh |
| B | New directory in `generated-reports/{vendor}/{YYYY}/{MM}/` | Run meta-prompt in a model with MCP tools active; write output to the path in shard `04-output-filename-pathing.md`; shard it with `scripts/shard-report.sh` |
| C | Sharded report | Use Edit on individual shards; for structural changes (section order, cross-refs), reconstruct first, edit, re-shard |
| D | — | Human-only gate |
| E | `site/data/reports/{vendor}/*.ts` | Update TS with latest data; run `bun run check:sync` from `site/` |
| F | Vercel deployment | Auto on push to `main` (no custom workflow) |
| G | `scripts/rss-monitor/` | Re-enable the disabled workflow to get update signals as GitHub Issues |

## Core Commands

Run from repo root unless stated otherwise.

### Sharding
```bash
# Split a flat .md into shards (verifies byte-identity automatically)
scripts/shard-report.sh path/to/flat.md path/to/flat-dir

# Reconstruct a sharded directory back into a flat file
scripts/reconstruct-report.sh path/to/flat-dir                    # stdout
scripts/reconstruct-report.sh path/to/flat-dir /tmp/rebuilt.md    # file
```

### Drift detection
```bash
# From site/, check that markdown reflects TS data
cd site && bun run check:sync               # both reports
cd site && bun run check:sync vercel-aws    # one vendor
```

Exit code 1 = hard error (e.g. internal version drift inside TS). Exit code 0 with warnings = expected soft drift (markdown prose not an exact substring match). Humans/agents triage warnings.

### Site
```bash
cd site
bun install        # first time
bun dev            # local dev (turbopack)
bun run build      # production build
bun run check      # ultracite lint
bun run check:sync # drift check
```

## Site ↔ Markdown Sync — How to Actually Do It

**When you edit site TS data:**

1. Make the TS edit in `site/data/reports/{vendor}/{section}.ts`.
2. Run `bun run check:sync` from `site/`. It will flag values that no longer appear in the markdown.
3. For each flagged value, open the corresponding shard and update the markdown prose. Shard-to-TS mapping is in `references/ts-to-markdown-map.md`.
4. Bump `metadata.version` **and** `footer.reportVersion` together — they must match. Add a changelog entry in the report's last shard.
5. Commit site + markdown together.

**When markdown changes first (e.g. a fresh AI-generated report in step B):**

1. Reconcile the new markdown facts into site TS files (this is the expensive manual step).
2. Re-run `bun run check:sync` until clean.
3. Regenerate any downstream narrative in markdown that now references the updated TS values.

**Classification of site TS files — which are auto-syncable:**

- **PURE MIRROR** (safe to auto-sync, markdown copy verbatim): `code.ts`, `deployment.ts`, `adoption.ts`, `delta.ts`
- **ENRICHED** (markdown data + site-only fields): `infrastructure.ts`, `pricing.ts`, `regions.ts`, `metadata.ts`, `hero.ts`, `footer.ts`
- **SITE-ONLY** (never touch markdown): `sections.ts`, `index.ts`, `report-schema.ts`

Full field-by-field classification: `references/site-ts-classification.md`.

## Known Gotchas

### 1. AWS-centric field names in shared schema
`FooterData.analyzedVersions` has fields named `aiSdk`, `strands`, `agentcore`. The Azure report reuses these keys for `agent-framework@1.0.1` and `azure-ai-projects@2.1.0` — semantically wrong but structurally fine. A proper fix (rename to `framework` / `infrastructure`) would touch `site/data/report-schema.ts` and every `footer.ts`. Treat as known debt; document, don't silently "fix" during unrelated work.

### 2. Version must be consistent within a report
`metadata.version` and `footer.reportVersion` must match. `check:sync` flags this as a hard error. When bumping, edit both.

### 3. Largest meta-prompt shard is still chunky
`meta-research-prompts/{Vendor}-Base-Research-Prompt/10-validated-citations-reference-data-mcp-sourced.md` runs 380–520 lines. Reference data is read atomically (not line-edited), so it's acceptable. If you need to edit individual citations, do a targeted `Edit` rather than reading the whole file.

### 4. RSS monitor is disabled by default
`.github/workflows/rss-monitor.yml.disabled` will not run as-is. Rename to `rss-monitor.yml` to enable. Requires `GITHUB_TOKEN` with `issues: write`.

### 5. Azure pricing pages are SPAs
Per `meta-research-prompts/Vercel-Azure-Base-Research-Prompt/08-technical-constraints.md`: static HTML fetches of `azure.microsoft.com/pricing/*` return `$-` placeholders. Agents must use MCP docs or browser automation (`agent-browser` skill). If a price cannot be verified, mark it as a DOCUMENTATION GAP; do not guess.

### 6. Reports at or below 500 lines stay flat
The two January 2026 AWS reports (`2026-01-08-…Opus-4.5.md` 939 lines, `…GPT-5.2.md` 536 lines) are intentionally not sharded. Only shard when the file crosses an editability threshold.

## Adding a New Platform Comparison (e.g. Vercel vs GCP)

Checklist:

1. **Meta-prompt**: Copy `meta-research-prompts/Vercel-AWS-Base-Research-Prompt/` → `Vercel-GCP-Base-Research-Prompt/`. Rewrite vendor-specific sections (ROLE, ARCHITECTURAL CLARIFICATION, MCP TOOLCHAIN, OUTPUT FILENAME & PATHING, VALIDATED CITATIONS). Keep the 6 REQUIRED SECTIONS structure.
2. **Run generation**: Execute the prompt in a model with the required MCP tools. Save output to `generated-reports/vercel-gcp/{YYYY}/{MM}/{YYYY-MM-DD}-Agent-Comparison-Report-{MODEL}.md`, then shard it.
3. **Schema**: Check `site/data/report-schema.ts`. If GCP-specific region flags differ from `AWSAgentCoreFeatures`/`AzureFoundryFeatures`, add a new interface and a parallel `GCPRegionalData` type. Follow the pattern established by Azure.
4. **Site data**: Create `site/data/reports/vercel-gcp/` with all 12 TS files. Start by copying a vendor directory whose data shape is closest.
5. **Registry**: Add to `site/data/reports/index.ts`:
   ```ts
   import { vercelGcpReport } from "./vercel-gcp/index.ts";
   export const reports = { ..., "vercel-gcp": vercelGcpReport } as const;
   ```
6. **Routing**: Add `site/app/reports/vercel-gcp/page.tsx` (copy from an existing vendor page).
7. **Platform colors/icons**: Add a `Platform` entry with `id: "gcp"` in the shared types if missing.
8. **RSS monitor**: Add a `src/reports/vercel-gcp.ts` config in `scripts/rss-monitor/`. Use `scripts/rss-monitor/prompts/update-platform-keywords.md` to generate the keyword list.
9. **README**: Add a row to the comparisons table.
10. **Drift check**: Run `cd site && bun run check:sync vercel-gcp` before committing.

## MCP Tools Required for Report Generation

Full list in `meta-research-prompts/{Vendor}-Base-Research-Prompt/03-mcp-toolchain-utilization-mandatory.md`. All four are MANDATORY per the prompt:

1. **GitHub MCP** — SDK release tags, repo metadata, issue ratios
2. **Platform docs MCP** — AWS Documentation MCP for AWS; reused with `learn.microsoft.com` URLs for Azure
3. **Vercel MCP** — AI Gateway catalog, FDT pricing, Sandbox/Workflow pricing
4. **Context7** — canonical code examples on latest SDK tags

If any of these are unavailable, flag it in the report's DOCUMENTATION GAPS section rather than falling back to model memory.

## Refreshing an Existing Report (Quarterly / After Major Announcements)

Use the self-refresh prompt embedded in each meta-prompt: shard `15-appendix-b-self-update-prompt.md` + `16-task-refresh-the-agent-comparison-master-prompt.md`. It contains a checklist for:

- Version updates (SDK release tags)
- Pricing validation (per-service table)
- Feature parity check (new announcements on both platforms)
- Blessed path validation (has either platform renamed/deprecated key APIs?)
- Regional availability verification
- Documentation link validation (spot-check 5 URLs per platform)

After refresh, update `APPENDIX A: PROMPT CHANGELOG` (shard 14) with today's date and a summary of changes.

## Anti-Patterns

- **Editing markdown to override site data.** Site wins; this is explicit in every report footer.
- **Regenerating a flat .md file after sharding.** The shards are the new source of truth for markdown. Reconstruction is for one-off reads only.
- **Splitting a shard that's already small (<300 lines).** Keep the section boundary that matches the `##` heading.
- **Adding subjective ratings (e.g. "7/10", "excellent").** Explicitly banned by every meta-prompt. Hard facts only.
- **Hallucinating prices, region counts, or version numbers.** Every fact must trace to an MCP-sourced or web-cited URL. Flag unknowns in DOCUMENTATION GAPS.
- **Bumping `metadata.version` without bumping `footer.reportVersion`.** `check:sync` will fail.
- **Committing a report without running `check:sync`.** It's one command and catches version drift instantly.
